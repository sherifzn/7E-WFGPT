# Codex Operating Contract

## Mission

Build a secure, configurable enterprise workflow platform that supports:

- visual workflow design
- reusable workflow templates and subprocesses
- Business Rules Engine / DMN integration
- human tasks and parallel branches
- RBAC plus attribute-based authorization
- team structures, team heads, deputies, and delegation of authority
- weighted assignment and workload balancing
- SLA, alerts, escalation, and emergency reassignment
- prerequisite detection and child-workflow triggering
- governed knowledge extraction from completed workflows
- AI-assisted design and selective runtime AI
- progressive decision autonomy with human override and auditability

## Non-negotiable architecture rules

1. Do not build the platform on .NET.
2. Do not couple the architecture to Codex or any coding-agent provider.
3. Do not allow direct access from workflows or AI agents to Oracle tables.
4. Oracle access must go through approved, typed connector interfaces.
5. Business thresholds and routing decisions belong in the BRE/DMN layer, not in application code.
6. Workflow execution must be durable, versioned, and recoverable.
7. Every workflow, rule, form, subprocess, connector, policy, and AI configuration must be versioned.
8. Separate the control plane from the runtime plane.
9. Separate workflow runtime data from analytics and reporting data.
10. Separate raw workflow history from approved knowledge.
11. No production deployment or production credential usage.
12. No secrets, personal data, legal evidence, financial records, or identity documents in source control, prompts, logs, fixtures, or tests.
13. Every externally visible operation must enforce authorization.
14. Every state-changing operation must produce an immutable audit event.
15. Every autonomous decision must pass through a Decision Policy Gateway.
16. AI output is untrusted input and must be schema-validated.
17. A failed AI task must have a deterministic fallback, normally human review.
18. Do not introduce a new dependency without documenting purpose, license, security impact, and operational impact.
19. Prefer modular architecture and explicit contracts over premature microservices.
20. Keep cloud and on-premises deployment portable until a hosting decision is approved.

## Work method

For every task:

1. Read this file and relevant architecture documents.
2. Restate the bounded scope in the pull-request description.
3. Identify affected modules and contracts.
4. Implement the smallest complete change.
5. Add or update tests.
6. Run formatting, linting, unit tests, integration tests, and security checks.
7. Document assumptions and unresolved decisions.
8. Do not make unrelated changes.
9. Do not merge or deploy.
10. Stop when an architecture decision is required and create an ADR proposal instead of guessing.

## Required pull-request content

- Problem statement
- Scope
- Architecture impact
- Security and privacy impact
- Data classification impact
- API or event-contract changes
- Tests added
- Migration implications
- Rollback approach
- Known limitations
- Follow-up work

## Initial implementation constraints

Until ADRs are approved:

- Use Kotlin or Java for core backend services.
- Use TypeScript and React for portals and the process designer.
- Use Python only for specialized AI, document, retrieval, and evaluation services.
- Use PostgreSQL for local platform development only.
- Use mocked Oracle, identity, email, and document connectors.
- Keep messaging, workflow engine, BRE/DMN engine, model provider, and hosting vendor behind interfaces.
- Use synthetic test data only.

## Forbidden actions

- No direct commits to protected branches.
- No disabling security controls to make tests pass.
- No suppressing failing tests without documented approval.
- No raw SQL generated by an AI agent against Oracle.
- No hard-coded user names, team names, monetary thresholds, or SLA values in domain code.
- No logging of tokens, passwords, access credentials, complete documents, or sensitive payloads.
- No automatic decision based only on similarity to historical cases.
